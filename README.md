Учебный проект подключение очереди и асинхронной обработки.

### Задание 1.
Оценить необходимость асинхронной обработки
## Асинхронная обработка видеоконвейера

Асинхронный подход здесь оправдан по двум причинам: стадии обработки
естественно независимы друг от друга, и сами кадры — тоже независимые
единицы данных.

В последовательной реализации каждый кадр полностью проходит все этапы —
`grayscale → детекция → мозаика` — прежде чем начнётся следующий.
Ресурсы при этом простаивают: пока работает одна стадия, остальные ждут.

Асинхронная архитектура превращает это в **конвейер**: пока один воркер
выполняет `detectMultiScale` для текущего кадра, другой уже делает
`cvtColor` для следующего, а третий применяет мозаику к предыдущему.
Система выигрывает не за счёт ускорения одного кадра, а за счёт роста
**пропускной способности**.

### Оценка выигрыша (метод HDD)

Предположим что примерное время обработки такое:

| Этап               | Время                |
|--------------------|----------------------|
| `cvtColor`         | ~5 мс                |
| `detectMultiScale` | ~40 мс               |
| `apply_mosaic`     | ~10 мс               |
| **Последовательно**| **~55 мс (~18 FPS)** |

 
Конвейеризация - если обрабатывается все параллельно, скорость обработки 
задаёт уже самое медленное звено — `detectMultiScale` 40 мс, что даёт 
выигрыш **~25 FPS** (ускорение **~1.4**).

Дополнительный выигрыш достигается за счёт устранения узкого места: 
если запустить несколько воркеров для стадии 
detectMultiScale (например, 2), её эффективное время уменьшается 
примерно до **20 мс**. Тогда пропускная способность конвейера может 
вырасти до **50 FPS**, что даёт уже порядка **2.5–3x** ускорения 
относительно исходного варианта.

**Итого**
Асинхронная обработка даёт заметный прирост производительности и лучше 
использует ресурсы системы, особенно при потоковой обработке видео. 
Однако она усложняет архитектуру (очереди, сериализация, 
обработка ошибок), поэтому оправдана в серверных или нагруженных 
сценариях, но избыточна для простых локальных задач.



### Задание 2. 

Выбрать брокер очередей, спроектировать асинхронный API для 
модели


Для этой задачи я бы выбрал RabbitMQ.
Обоснование: у нас не потоковая аналитика уровня “миллионы событий и 
хранение истории”, а асинхронный ML-конвейер из нескольких стадий:
```
frame → grayscale → face detection → mosaic → output frame
```

Здесь важнее:
гарантированно передать кадр между стадиями;
обработать сообщение один раз или повторить при ошибке;
удобно развести этапы по очередям;
не усложнять архитектуру.

Kafka сильна, когда нужен большой поток событий, хранение истории, replay, 
горизонтальное масштабирование consumer groups. Для видео-конвейера это может быть избыточно.

Redis Pub/Sub очень простой, но классический Pub/Sub не хранит сообщения для 
отключенных подписчиков. Если воркер упал, событие можно потерять. Redis Streams
надежнее, но тогда это уже отдельная модель работы.

RabbitMQ лучше подходит под задачу “очередь задач между воркерами”: есть очереди, 
acknowledgements, retry-поведение, routing, dead-letter queue. Поэтому для учебного 
и прикладного ML-конвейера он наиболее оправдан. FastStream поддерживает RabbitMQ, 
Kafka, Redis и умеет генерировать AsyncAPI-документацию для брокерных приложений.

Как действуем

Каждая стадия конвейера оформляется как отдельное событие:

#### 1. `frame.received`
- **Вход:** исходный кадр
- **Выход:** `grayscale.frame.created`
- **Операция:** `cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)`

#### 2. `grayscale.frame.created`
- **Вход:** серый кадр
- **Выход:** `faces.detected`
- **Операция:** `face_cascade.detectMultiScale(gray, 1.1, 4)`

#### 3. `faces.detected`
- **Вход:** исходный кадр + координаты лиц
- **Выход:** `frame.mosaiced`
- **Операция:** `apply_mosaic_effect(face_roi, pixel_size=30)`

между событиями лучше передавать не сам numpy.ndarray напрямую, 
а сериализованное представление кадра, например JPEG/PNG в base64. 
Это важно чтобы при пересылке не испортить исходное изображение
т.к. некоторые байты, могут быть при передаче неверно интерпертированны и 
искажены.

Для реального production лучше хранить кадры в S3/MinIO/файловом хранилище, 
а через брокер передавать только ссылку.

### Задание 3. 
Выбрать стратегию (паттерн) загрузки моделей и реализовать ее

В данной реализации используется Strategy Pattern, который позволяет 
отделить способ загрузки модели от логики её использования. 
Класс FaceDetector отвечает только за детекцию лиц, а конкретная 
стратегия (LazyLoadingStrategy или WarmLoadingStrategy) определяет, 
когда именно загружается модель.

Для нашего примера, на мой взгляд можно использовать два 
варианта:
Основной, Warm loading — модель загружается заранее при старте. 
Т.к. может использоваться постоянно, в том числе online и важно 
избежать задержки на первом кадре. 

А вот для offline-сценариев, более уместен Lazy loading, когда модель 
может использоваться редко.

Реализация паттернов

### Задание 4
Создать загрузку данных батчами в модель

Реализовали класс BatchProcessor, который накапливает кадры в буфере и 
запускает обработку при заполнении буфера или истечении таймаута. 
Внутри flush() кадры всё равно обрабатываются последовательно в цикле — 
никакой параллельности нет. Причина в том, что 
`cv2.CascadeClassifier.detectMultiScale` принимает один кадр и не 
поддерживает batch-inference. Накопить 64 
кадра и передать их пачкой в модель невозможно — архитектурно паттерн есть, 
а реального выигрыша нет. Более того, батчинг добавляет латентность: 
первый кадр ждёт 63 следующих, прежде чем получит результат. 
Для видеопотока это регрессия, а не оптимизация. 
Честная реализация для Haar Cascade — просто функция на три строки 
без какого-либо буфера. Паттерн BatchProcessor имеет практический смысл 
только при замене детектора на модель с настоящим batch-inference, что-то типа 
ONNX Runtime или PyTorch, где один вызов обрабатывает тензор (N, C, H, W) целиком 
на GPU.

Реализация паттерна
📄 [batch_processor.py](src/batch_processor.py)
